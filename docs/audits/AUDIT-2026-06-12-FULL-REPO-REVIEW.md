### F-001: CI has never been green: every run in repo history (9/9) failed; current cause is MSVC-rejected constexpr validators

- **Severity:** critical
- **Dimension:** Git hygiene & CI
- **Location:** `include/blackhole_ds/units.hpp (line ~188)`
- **Status:** OPEN

gh run list shows all 9 CI runs on main since the baseline commit (2026-05-26) concluded 'failure', including the latest (Iteration 8, run 27293174242). The latest run fails at 'CMake build' with MSVC error C3615 at units.hpp lines 188, 195, 203: photon_sphere_radius_valid, isco_radius_valid, shadow_diameter_valid are declared constexpr but call std::abs, which is not constexpr until C++23 (and the project pins C++20). Iterations 1-7 runs also all failed at their 'Build' step, so this is not a one-off regression.

### F-002: Shadow 'diameter' is actually the shadow RADIUS — factor-of-2 error in a headline observable

- **Dimension:** Physics & mathematics
- **Location:** `include/blackhole_ds/units.hpp (line ~203)`
- **Status:** OPEN

The repo defines r_g = GM/c^2 (units.hpp:140), so every quantity named 'shadow_diameter' carrying 5.196 is half the true diameter. **Suggested fix:** Either rename everything to shadow_radius_rg (and document diameter = 2x), or keep 'diameter' and change the baseline to 2*sqrt(27) = 10.392 M. Update units.hpp validator + comments, harness function, schema.sql column comment, smoke_tests.cpp expectation, and the '~5.2 rg' phrasing in docs/INSPIRATION_BRIEFING.md:18 and docs/audits/INITIAL_250_POINT_GOLD_STANDARD_AUDIT.md:21 to '~10.4 GM/c^2 diameter (~5.2 Schwarzschild radii)'.

### F-004: CSV export prints doubles at default 6-significant-figure precision, silently lossy for a project that validates to 1e-12

- **Severity:** major
- **Dimension:** C++ code
- **Location:** `include/blackhole_ds/data/csv_writer.hpp (line ~29)`
- **Status:** OPEN

write_row streams `mass_solar`, `spin_a_over_M`, and `value_si_meters` straight to the ostream with no precision control (lines 29-33). Default std::ostream precision is 6 significant digits. For mass_solar=1.0, schwarzschild_radius_m = 2*1476.625038 = 2953.250076 m is exported as '2953.25', dropping ~7 digits. The same writer is the project's only data-export path and the headers claim 1e-12 analytic tolerance (units.hpp:191, validators) - the CSV cannot represent values to anywhere near that. Spin values like a=-0.998+1.996*i/9 also round. This makes round-trip/regression comparison against the 1e-12 validators impossible from the exported file.

**Suggested fix:** In write_row (or the CsvWriter ctor) set full round-trippable precision before writing numeric fields, e.g. `out_ << std::setprecision(std::numeric_limits<double>::max_digits10)` (17) or use std::format("{:.17g}", v). Save/restore stream flags so callers are unaffected.

**Verifier confirmation:** Verified directly in source. csv_writer.hpp:29-33 streams all three doubles with no precision control, and the sole caller (src/cli/main.cpp emit_csv, line 158) passes std::cout in default state — the setprecision(3)/(4) calls at main.cpp:134/144 are in the emit_text branch which never executes for --format csv. Not critical (in-memory values and validators are correct; nothing is computed wrong), but more than minor: a silent precision loss in the only export path that defeats the project's stated data-integrity contract. Major stands.

### F-006: units.hpp over-claims: no binary operator+/- exists, so 'r + t is a compile error' is misleading and even same-dimension addition won't compile

- **Severity:** major
- **Dimension:** C++ code
- **Location:** `include/blackhole_ds/units.hpp (line ~63)`
- **Status:** OPEN

The header's banner (line 5) and Usage Contract (lines 236-240) advertise that `auto nonsense = r + t;` is a COMPILE ERROR 'exactly what we want', implying that same-dimension addition (`r1 + r2`) DOES work. But `Quantity` defines only compound `operator+=` / `operator-=` (lines 63-78) and a unary `operator-` (line 80). There is no free or member binary `operator+`/`operator-` anywhere in the header (confirmed by grep). Consequence: `r + t` does not compile, but neither does `r1 + r2` for two same-dimension Lengths, nor `e1 - e2` for Energies. The 'mixing dimensions is the only thing that fails' story is false: ALL binary +/- fails. Any future integrator that writes `pos = pos + vel*dt` using these types will not compile, and the documented example is aspirational, not real. The type system under-delivers (no usable addition) while the comments over-claim (selective compile error).

**Suggested fix:** Either (a) add constrained free functions `template<class Tag,class Rep> constexpr Quantity<Tag,Rep> operator+(Quantity<Tag,Rep> a, Quantity<Tag,Rep> b)` and the matching `operator-` (which by construction only accept identical Tag, giving exactly the 'same-dim adds, cross-dim fails' semantics the comment claims), or (b) correct the header/Usage-Contract comments to state that only `+=`/`-=` exist and binary `+`/`-` are intentionally omitted. Then add a static_assert/SFINAE test that `r1 + r2` compiles and `r + t` does not.

**Verifier confirmation:** Verified directly against include/blackhole_ds/units.hpp. Line 5 banner and Usage Contract lines 237-240 advertise selective dimensional safety ('auto nonsense = r + t; // COMPILE ERROR — exactly what we want'), implying same-dimension binary addition works. But the Quantity class defines only compound operator+=/-= (lines 63-70), scalar *=//= (71-78), unary operator- (80), comparisons, and one cross-dimension operator/(Length,Time) (147). A repo-wide grep over all C++ sources confirms no binary operator+ or binary operator- exists anywhere; the constructor is explicit with no conversion to Rep, so no fallback path exists. Hence r + t fails to compile, but so does r1 + r2 for two Lengths — the compile error stems from total absence of binary additive operators, not dimensional checking. The 'mixing dimensions is the only thing that fails' story is false for binary ops. Current consumers (tests/smoke_tests.cpp, metrics headers) avoid binary +/- so the build is green — ruling out critical — but CONTRIBUTING.md, the Scientific Integrity Charter, and the header's own Usage Contract mandate these types for every future integrator, so the first 'pos = pos + step' will fail (no binary operator* for scalar scaling exists either, slightly worse than the finding states). This is a misleading claim plus a latent bug in the mandated foundation layer: major per the rubric.
