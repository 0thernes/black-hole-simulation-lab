## §0 — Executive Framework

### 0.1 Purpose & Scope

This Standard Operating Procedure defines the **mandatory** code review and audit process for all software changes that affect production systems. It establishes a **35-dimension weighted framework** with floor thresholds, hard floors, anti-pattern registries, and automated enforcement gates that must pass before any code reaches production.

**In Scope**: Application code, infrastructure-as-code, configuration, database migrations, ML/AI model artifacts, API contracts, documentation-as-code, feature flags, deployment manifests, and prompt/engineering artifacts for AI systems.

**Out of Scope**: Vendor-managed SaaS configuration (unless custom integration code), one-off scripts with no production impact, personal prototypes not in any CI pipeline.

**Design Principles**:
- **Actionable**: Every dimension has explicit checklists, scoring rubrics, and anti-patterns
- **Measurable**: Numeric scores (0.0–10.0) with weighted aggregation and floor thresholds
- **Enforceable**: CI/CD pipeline gates map directly to dimension floors
- **Auditable**: Finding tracking, severity classification, and compliance mapping enable traceability
- **Calibratable**: Inter-rater reliability protocol ensures consistent scoring across reviewers

### 0.4 Scoring Methodology & Calibration

#### Scoring Convention

| Score Range | Rating | Emoji | Tier | Action |
|---|---|---|---|---|
| **9.0–10.0** | Elite | 🟢 | Tier 5 | Merge immediately; exemplary; share as pattern |
| **7.0–8.9** | Acceptable | 🟢 | Tier 3–4 | Merge with minor nits tracked in follow-up |
| **5.0–6.9** | Marginal | 🟡 | Tier 2 | Requires fixes before merge; remediation plan required |
| **3.0–4.9** | Major Deficiency | 🟠 | Tier 1 | Block merge; significant rework required |
| **0.0–2.9** | Critical Failure | 🔴 | Tier 0 | Block merge; immediate escalation; potential incident |

**Pass Threshold**: Every dimension ≥ its floor. Overall weighted average ≥ 7.5. Zero 🔴 unresolved. Zero 🟠 unresolved. All CI gates green. SLO compliance ≥ 99.9%.

#### Scoring Calculation

1. **Each dimension** is scored 0.0–10.0 by the reviewer using the rubric in §2
2. **Weighted score** = Σ (dimension_score × dimension_weight)
3. **Overall score** = Weighted score / 100 (normalized to 0–10 scale)
4. **Pass criteria**:
   - Overall weighted score ≥ **7.5**
   - **Every** dimension ≥ its pass floor
   - **Hard floor dimensions** (D01, D20) ≥ **8.0** — no exceptions
   - Zero 🔴 (Critical) or 🟠 (Major) findings unresolved
   - All CI/CD pipeline gates green

#### Inter-Rater Reliability Protocol

To ensure consistent scoring across reviewers:

| Calibration Activity | Frequency | Target | Method |
|---|---|---|---|
| **Reference PR Review** | Quarterly | κ ≥ 0.7 (Cohen's kappa) | All reviewers independently score the same PR; compare scores |
| **Score Distribution Audit** | Monthly | σ ≤ 1.0 across reviewers | Statistical analysis of score distributions per dimension |
| **Calibration Workshop** | Quarterly | Consensus on edge cases | Team discusses borderline scores; aligns on rubric interpretation |
| **New Reviewer Shadowing** | On hire | 3 shadow reviews before solo | New reviewer shadows 3 reviews, then is shadowed for 3 reviews |

### 0.6 Exception Process

When a dimension score falls below its floor, the following exception process applies:

1. **Document**: Author must file an exception request citing dimension, score, justification, and remediation timeline
2. **Review**: Tech Lead and relevant Champion review the exception
3. **Risk Acceptance**: Tech Lead formally accepts risk with a documented expiration date (max 30 days)
4. **Track**: Exception logged in finding tracker with owner and due date
5. **Remediate**: Must be resolved before exception expiration; no renewals without VP Eng approval

> **Hard Floor Exception**: D01 (Security) and D20 (Compliance) have **hard floors** — no exceptions permitted below 8.0.

**Exception Template**:

```markdown
## Exception Request

**Dimension**: D__
**Current Score**: __ / 10
**Floor**: __
**Justification**: [Why the floor cannot be met at this time]
**Risk Assessment**: [What risk does this create? Likelihood × Impact]
**Remediation Plan**: [Concrete steps to meet the floor]
**Target Date**: [When the floor will be met]
**Approved By**: __ (Tech Lead) __ (Champion)
**Expiration**: [Max 30 days from approval]
```

### 0.7 AI-Assisted Review Protocol

| Aspect | Requirement | Enforcement |
|---|---|---|
| **AI Code Generation Disclosure** | All AI-generated code must be tagged with `@ai-generated` annotation | Pre-commit hook |
| **AI Review Tool Usage** | AI review tools (CodeRabbit, GitHub Copilot Review, etc.) are **advisory only** — human reviewer retains final authority | Policy |
| **AI Hallucination Guard** | AI-suggested code must be verified for correctness; no blind acceptance | Reviewer checklist |
| **Prompt Injection Awareness** | Code that processes LLM output must sanitize against prompt injection | D01 checklist |
| **AI Attribution** | Commits with > 50% AI-generated lines must include `Co-authored-by: AI Assistant` in git message | CI gate |
| **AI Code Quality Bar** | AI-generated code must meet the **same** quality standards as human-written code — no exceptions | Policy |
| **Model Provenance** | AI tools used in code generation must be documented in PR metadata (model, version, provider) | PR template |
| **AI Output Validation** | LLM responses used in logic must be validated for type, range, format before use | D02 checklist |
| **AI Bias Check** | AI-generated content must be reviewed for bias, hallucination, and factual accuracy | D30 checklist |
| **AI Security Review** | AI-generated code must pass the same security review as human-written code | D01 checklist |

### D01 — Security & Threat Modeling (Weight: 7%, Floor: 8.0, Hard Floor: ✅)

#### Threat Model (STRIDE Analysis)

| Threat Type | Example | Mitigation | Score 0–10 |
|---|---|---|---|
| **Spoofing** | Session hijack, CSRF, credential replay, JWT tampering, AI prompt injection | SameSite cookies, CSRF tokens, MFA, short-lived tokens, JWT validation, prompt sanitization | |
| **Tampering** | XSS, DOM manipulation, prototype pollution, parameter injection, mass assignment, AI output manipulation | CSP, Trusted Types, input validation, frozen prototypes, allowlists, AI output sanitization | |
| **Repudiation** | Unlogged actions, missing audit trail, shared credentials, AI-generated code attribution gaps | Structured logging, immutable audit log, correlation IDs, individual accounts, AI attribution | |
| **Information Disclosure** | Console logging secrets, PII in URLs, error stack traces in prod, verbose error messages, AI model data leakage | Log redaction, HTTPS-only, custom error pages, source map restriction, model output filtering | |
| **Denial of Service** | ReDoS, event listener leak, unbounded memory, API rate bypass, hash collision, LLM token exhaustion | Rate limiting, input length caps, circuit breakers, resource quotas, hash randomization, token budgets | |
| **Elevation of Privilege** | IDOR, privilege escalation, insecure direct object references, JWT algorithm confusion, AI tool privilege escalation | RBAC, ABAC, principle of least privilege, resource-level auth checks, AI tool sandboxing | |

#### Attack Surface Inventory

| Attack Vector | Status | Mitigation | Verified |
|---|---|---|---|
| **Client-Side Injection** | | CSP, Trusted Types, sanitization | |
| **API Endpoints** | | Auth, rate limiting, input validation | |
| **Third-Party Scripts** | | SRI, Subresource Integrity, CSP | |
| **Local Storage** | | No sensitive data, encryption at rest | |
| **Service Workers** | | Scope restricted, no wildcard | |
| **PostMessage** | | Origin validation, structured clone | |
| **Web Workers** | | No DOM access, message validation | |
| **Supply Chain** | | Lockfile, audit, Snyk/Socket, SBOM | |
| **OAuth / OIDC Flows** | | PKCE, state parameter, nonce, redirect URI validation | |
| **WebSocket Connections** | | Origin validation, message validation, rate limiting | |
| **File Upload** | | Type validation, size limits, virus scan, sandboxed storage | |
| **DNS / Subdomain Takeover** | | CNAME validation, dangling DNS detection | |
| **AI/LLM Integration** | | Prompt injection guards, output sanitization, token budgets, model access controls | |
| **Container Escape** | | Seccomp, AppArmor, read-only filesystem, non-root user | |
| **Secrets in CI/CD** | | Vault/Sealed Secrets, no plaintext in pipelines, secret scanning | |

#### Security Checklist

- [ ] **No `eval()`, `new Function()`, `document.write()`, or `innerHTML` with untrusted data**
- [ ] All user inputs sanitized/escaped before DOM insertion or API calls
- [ ] No secrets, tokens, or PII in client-side code, logs, or source maps
- [ ] CSP headers configured; no `unsafe-inline` / `unsafe-eval` in production
- [ ] Dependencies audited; zero high/critical CVEs
- [ ] All API calls use HTTPS with proper error handling (no silent fail-open)
- [ ] Rate limiting and input validation on all endpoints
- [ ] Session management follows OWASP Top 10 (2025) guidelines
- [ ] No `javascript:` URIs or `data:` URIs with executable content
- [ ] Prototype pollution guards on all `Object.assign` / spread merges
- [ ] **STRIDE threat model completed for this change**
- [ ] **Attack surface inventory updated**
- [ ] **Penetration test results reviewed (if applicable)**
- [ ] **Secrets scanned (gitleaks, truffleHog) — no leaked credentials**
- [ ] **CORS policy reviewed — no wildcard origins in production**
- [ ] **Subresource Integrity (SRI) hashes on all CDN-hosted scripts**
- [ ] **Content-Security-Policy report-only mode tested before enforcement**
- [ ] **HTTP security headers verified**: X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy, Strict-Transport-Security
- [ ] **OAuth/OIDC: PKCE enabled, state parameter validated, redirect URI allowlisted**
- [ ] **WebSocket: origin validation, message schema validation, rate limiting**
- [ ] **File upload: type allowlist, size limits, virus scan, sandboxed storage**
- [ ] **JWT: algorithm allowlisted (HS256/RS256), expiry validated, issuer checked**
- [ ] **Zero-trust: mTLS between services, service mesh authentication, no implicit trust**
- [ ] **AI prompt injection: all LLM inputs sanitized, output validated before rendering**
- [ ] **Post-quantum readiness: crypto agility assessed; hybrid key exchange planned for NIST PQC standards (ML-KEM, ML-DSA)**
- [ ] **Container security: non-root user, read-only filesystem, seccomp profile, no privileged mode**
- [ ] **OWASP Top 10 (2025) all items addressed**

#### Anti-Patterns 🚫

| Anti-Pattern | Why It's Wrong | Fix |
|---|---|---|
| `innerHTML = userInput` | XSS vector | Use `textContent` or DOMPurify |
| `eval(jsonString)` | Code injection | Use `JSON.parse()` |
| Storing tokens in `localStorage` | Accessible to XSS | Use HttpOnly cookies |
| `*` in CORS `Access-Control-Allow-Origin` | Allows any origin | Specify exact origins |
| Shared service account credentials | No repudiation | Individual accounts + MFA |
| JWT with `algorithm: none` | Bypasses signature verification | Always verify algorithm |
| Hardcoded secrets in code | Exposed in VCS | Use Vault / Sealed Secrets |
| Trusting AI-generated code without review | Hallucinated vulnerabilities | Mandatory human review of AI output |
| Using MD5/SHA1 for security | Cryptographically broken | Use SHA-256+ or Argon2 for passwords |
| No mTLS between services | Implicit trust on internal network | Zero-trust: verify every request |

### D02 — Logic Correctness (Weight: 6%, Floor: 7.5)

| Sub-dimension | Critical Items | Score 0–10 |
|---|---|---|
| **Control Flow** | Dead code paths, unreachable branches, infinite loops, off-by-one errors | |
| **State Machine Integrity** | Invalid transitions, race conditions, stale state, orphan states | |
| **Boundary Conditions** | Edge cases (0, -0, NaN, Infinity, empty string, null, undefined), boundary indices | |
| **Concurrency** | Event ordering, promise chaining, async/await error propagation, debounce correctness | |
| **Data Integrity** | Type coercion traps (`==` vs `===`), floating-point precision, integer overflow | |
| **Business Logic** | Calculation correctness, formula validation, domain rule enforcement | |
| **Idempotency** | Operations safe to repeat; no double-submit; no side effects on retry | |
| **Temporal Logic** | Time zones, daylight saving, leap years, race conditions on timestamps | |
| **Null Safety** | Null/undefined guards, optional chaining, nullish coalescing, defensive defaults | |
| **Error Propagation** | Errors bubble correctly, no swallowed exceptions, proper error types | |
| **AI Output Validation** | LLM responses validated for type, range, format before use in logic | |

**Checklist:**

- [ ] All branches reachable and tested
- [ ] State transitions validated against explicit transition table
- [ ] Edge cases covered: zero, negative, NaN, Infinity, empty, null, undefined
- [ ] No floating-point comparison without epsilon tolerance
- [ ] Async operations properly sequenced; no unhandled promise rejections
- [ ] No `==` where `===` is intended
- [ ] All switch statements have default cases
- [ ] No silent failures — every error path has explicit handling or logging
- [ ] **Idempotency verified for all write operations**
- [ ] **Date/time operations use UTC or explicit timezone; no implicit local time**
- [ ] **Race conditions identified and guarded (mutex, atomic ops, or serialization)**
- [ ] **Null/undefined guards on all external data access points**
- [ ] **Error types are specific (not generic `Error`); include context and code**
- [ ] **No `catch (e) { }` empty blocks — at minimum log the error**
- [ ] **AI-generated logic verified independently; no blind trust in LLM output**
- [ ] **Property-based testing applied for critical business logic (fast-check, jqwik)**

#### Anti-Patterns 🚫

| Anti-Pattern | Why It's Wrong | Fix |
|---|---|---|
| `0.1 + 0.2 === 0.3` | Floating-point precision | Use epsilon: `Math.abs(a - b) < Number.EPSILON` |
| `catch (e) { /* empty */ }` | Swallowed errors | Log, rethrow, or handle explicitly |
| `new Date('2024-01-01')` | Timezone-dependent parsing | Use `Date.UTC()` or `Temporal` |
| `==` for comparison | Type coercion surprises | Always use `===` |
| Mutable shared state | Race conditions | Immutable data, atomic operations |
| Trusting AI output without validation | Hallucinated logic | Validate type, range, format before use |

## §6 — Review Process SOP

### Step 1: Pre-Review Self-Check
Author completes the self-review checklist for all applicable dimensions before requesting review. Use the dimension checklists in §2 as the self-audit template.

### Step 2: Automated Gate Check
CI pipeline runs all pre-merge gates (§5.1). PR cannot be merged if any blocking gate fails. Advisory gates (AI code review) provide suggestions but do not block.

### Step 3: AI-Assisted Review (Advisory)
AI review tools (CodeRabbit, GitHub Copilot Review, etc.) provide advisory comments. Human reviewer retains final authority. AI suggestions must be verified — no blind acceptance.

### Step 4: Human Peer Review
Minimum required reviewers based on review type (see §0.5). Reviewer scores each applicable dimension using the rubric in §2.

### Step 5: Security & Compliance Review
Security Champion reviews D01, D18, D20, D31, D32 for all Full and Security reviews. Required for any change touching auth, data handling, crypto, or AI models.

### Step 6: Architecture Review (if applicable)
Tech Lead reviews D13, D22, D26 for major architectural changes. Required for new services, database changes, or API contract changes.

### Step 7: Final Approval & Merge
Tech Lead gives final approval. All blocking gates must be green. Exceptions must be documented per §0.6. Score card is posted in the PR.

| **Likely** (60–90%) | 🔴 EXTREME | 🟠 HIGH | 🟡 MEDIUM | 🟢 LOW |
| **Possible** (30–60%) | 🟠 HIGH | 🟡 MEDIUM | 🟢 LOW | 🟢 LOW |
| **Unlikely** (10–30%) | 🟡 MEDIUM | 🟢 LOW | 🟢 LOW | 🔵 NEGLIGIBLE |
| **Rare** (< 10%) | 🟢 LOW | 🟢 LOW | 🔵 NEGLIGIBLE | 🔵 NEGLIGIBLE |

### 10.2 Risk Response Matrix

| Risk Level | Response | SLA | Escalation | Example |
|---|---|---|---|---|
| 🔴 **EXTREME** | Immediate fix; block all deployments | < 4 hours | VP Engineering + Security | Active data breach, RCE vulnerability |
| 🟠 **HIGH** | Fix before next release; block merge | < 24 hours | Tech Lead + Security | Auth bypass, SEV2 outage |
| 🟡 **MEDIUM** | Schedule fix in current sprint | < 1 sprint | Reviewer | Missing test, slow query |
| 🟢 **LOW** | Backlog; address when convenient | Backlog | N/A | Code style, minor optimization |
| 🔵 **NEGLIGIBLE** | Accept; no action required | N/A | N/A | Cosmetic issue |

## §13 — Reviewer Certification & Training

### 13.1 Reviewer Tiers

| Tier | Requirements | Privileges |
|---|---|---|
| **Junior Reviewer** | Completed onboarding; 3 shadow reviews; passed calibration quiz | Can review Standard and A11Y reviews |
| **Senior Reviewer** | 50+ reviews; inter-rater κ ≥ 0.7; no SEV1 caused by missed review | Can review all types; can mentor junior reviewers |
| **Security Champion** | Senior reviewer + security training (OWASP Top 10, STRIDE); pen test awareness | Required for Security reviews; can approve D01 exceptions |
| **A11Y Champion** | Senior reviewer + A11Y training (WCAG 2.2, screen reader testing) | Required for A11Y reviews; can approve D04 exceptions |
| **Architecture Reviewer** | Senior reviewer + architecture training (C4 model, ADRs) | Required for architecture reviews; can approve D13 exceptions |

### 13.2 Calibration Protocol

| Activity | Frequency | Participants | Target |
|---|---|---|---|
| **Reference PR Review** | Quarterly | All reviewers | κ ≥ 0.7 (Cohen's kappa) |
| **Score Distribution Audit** | Monthly | Tech Lead | σ ≤ 1.0 across reviewers |
| **Calibration Workshop** | Quarterly | All reviewers | Consensus on edge cases |
| **New Reviewer Shadowing** | On hire | New + Senior reviewer | 3 shadow + 3 shadowed reviews |

### 13.3 Reviewer Onboarding Checklist

- [ ] Read this SOP in full
- [ ] Complete 3 shadow reviews with a Senior Reviewer
- [ ] Complete 3 reviews shadowed by a Senior Reviewer
- [ ] Pass the calibration quiz (≥ 80% score)
- [ ] Complete security awareness training (OWASP Top 10)
- [ ] Complete A11Y awareness training (WCAG 2.2 basics)
- [ ] Set up all required tools (ESLint, axe-core, Lighthouse, etc.)
- [ ] Review the anti-pattern registry (§12)
- [ ] Review the compliance mapping (§9) for applicable regulations
